# 長廊景深與展場漫步 — Phase F／G 提案與實現限制

> **載入時機**：討論 Phase F（長廊景深）、Phase G（展場漫步彩蛋）、或任何
> 「作品牆要更有展場感」的視覺提案時。
> **狀態**：提案，方向未定（見 `prepare.md` 待討論事項 #2）。定案後轉 `MR-014`，本檔改記落地細節。

---

## 起點

2026-07-24 用戶提出一張參考圖：第一人稱視角的展場走廊，作品掛在左右牆上沿透視後退，
畫面有消失點與地板網格，左上角資訊面板寫 `ON VIEW` + 襯線斜體作品名 + 等寬小字說明
（媒材／尺寸／年份），底部是 `← BACK` / `WALK ON →`，提示 `ARROW KEYS TO WALK`。

這不是新方向——`prepare.md` 的 Phase 重排已經排了 **Phase F（長廊景深＋詳情頁 pan/zoom，
純視圖層）** 與 **Phase G（展場漫步彩蛋，須待 E／F 穩定）**。本檔是把那兩格填上內容。

---

## 關鍵拆解：質感有八成不來自 3D

把參考圖的元素逐項歸因，會發現多數「高級感」是平面設計給的：

| 元素 | 來源 | 需要 3D？ |
|------|------|-----------|
| 襯線斜體作品名 | 字體搭配 | ✗ |
| `ON VIEW` 紅色小標、等寬大寫說明 | 排版層級 | ✗ |
| 白邊裱框（mat + 細框） | `border` / `padding` | ✗ |
| 作品浮離牆面的投影 | `box-shadow` | ✗ |
| 牆面展籤（作品名等寬大寫置中於卡片下方） | 一行文字 | ✗ |
| 左側大色塊疊印的資訊面板 | `mix-blend-mode` | ✗ |
| 消失點透視、可以走進去 | CSS 3D transform | ✓ |
| `ARROW KEYS TO WALK` / `WALK ON` | 場景狀態機 | ✓ |

**這個歸因直接決定投報率**：前六項沒有任何一條會踩到下方的限制，最後兩項則是全部限制的來源。

---

## Phase G 的技術路徑（若要做第一人稱）

不需要 Three.js 或任何 3D 函式庫，純 CSS 3D 即可：

- 場景容器 `perspective`，左右牆 `rotateY(±90deg)`，地板 `rotateX(90deg)`
- 每件作品是**普通 DOM 元素**，只是 `translateZ(-n × 間距)` 掛在牆上
  → 圖片、點擊、無障礙樹、IndexedDB 取圖全部沿用現況，不需為 3D 另建一套
- 「走」＝改場景上的一個 CSS 變數（相機 Z）並補間
- **離散步進**（一次一件，配 `BACK` / `WALK ON`）而非自由漫遊——參考圖本身就是這個設計，
  也剛好避開自製遊戲引擎的坑

---

## 實現限制（做 Phase G 前必須先接受這幾條）

**1. Safari 的 `preserve-3d` 遇祖先 `overflow` 會整個壓平**
現有長廊 `WorkRail.vue:161-164` 是 `overflow-x: auto` + `scroll-snap`，天生帶 overflow。
→ 走廊**不能疊進現有長廊**，必須是獨立的第三種瀏覽模式。

**2. 手機基本上不成立**
`<900px` 現在走網格（`ARCHITECTURE.md` 版面尺寸表）。第一人稱在手機視野裡一次只看得到一件，
等於退化成 slideshow。MR-008 已因近似理由棄選過平板兩欄。
→ 要嘛手機退回網格（多維護一套模式），要嘛接受手機體驗明顯變差。

**3. 30 秒法則——最硬的一條**
第一人稱代表訪客**必須走**才看得到第 5 件作品，招募端不會走。MR-001 的結論是
「遊戲層必須是加分而非路障」。
→ 走廊只能是第三顆模式鈕（依媒材／依展覽／走進展場），**不可為預設入口**。
這正是 `prepare.md` 把它定為「Phase G 彩蛋」而非主線的原因。

**4. 近景圖片會糊**
`utils/image.ts` 的 `THUMB_MAX_EDGE = 800`、`VIEW_MAX_EDGE = 1800`（MR-009）。
走廊裡最近那件會佔到半個畫面寬，800px 撐不住。
→ 近景改抓 1800px 那張（多一次大圖載入），或接受略糊。這是實打實的成本。

**5. `prefers-reduced-motion` 下走廊沒有意義**
走廊的價值全在位移補間。開了減少動態就該**直接不提供**這個模式，
而不是給一個瞬移版本（`main.css:100` 已關掉全站動畫與轉場）。

**6. 襯線字是唯一的新依賴風險**
`main.css:34-35` 目前只有 `--font-sans` / `--font-mono`，全系統字、零 `@font-face`。
加襯線要嘛用系統襯線（各平台長相不一），要嘛自帶 web font（100–200KB 首屏成本）。

**7. 好消息：不動資料層**
沿用現有 `filteredWorks`；`aspect-ratio` 那套版位邏輯在 3D 下依然成立
（transform 不改 layout box，`ARCHITECTURE.md` 的「版面尺寸唯一真相」不受影響）。
且**展覽動線放進走廊特別合理**——策展本就是有序的（MR-012 ①），走廊即那條動線的具象化。

## 建議：拆兩步，先拿便宜的八成

**Phase F 先做，不含第一人稱**：裱框白邊、牆面投影、展籤、`ON VIEW` 標籤、襯線標題、
卡片沿 Z 軸微幅後退的景深。上述七條限制**一條都不會踩到**——不動 overflow、手機照樣網格、
招募端一眼看完、圖片不必換解析度。預期能拿到參考圖七八成的質感。

**Phase G 再做真走廊**，作為第三種瀏覽模式，帶著限制 1–6 去設計。
